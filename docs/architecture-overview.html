skill-drift, from the helicopter

A local macOS menu-bar app that flags when your installed Claude Code Skills have drifted Behind their GitHub repos. This is the mental model — what Tauri is, how the pieces fit, and what's actually built vs. still on paper.

1 · What even is Tauri?

Two processes that talk over a bridge. That's the whole idea.

The webview is just a headless browser. Same React you'd ship to the web — except it can phone the Rust side. Two doorways across the bridge: the webview calls commands with invoke("name", args), and Rust fires events the webview listens for. Nothing else crosses.

2 · The one big decision: keep the brain in TypeScript

You're fluent in TS, new to Rust. So the architecture is drawn so Rust never decides anything that matters. Every "is this Skill Behind?" judgment happens in TS — where the engine already exists, fully tested, and where you can review it. Rust is hands, not brain. (ADR-0002, "the C3 seam".)

TypeScript — owns all logic + view
• read & interpret the Manifest
• poll GitHub, compare hashes
• classify Behind / Current / Removed / …
• build the menu the user sees
↳ has a real test suite (878 lines)
Rust — thin native shell only
• draw the tray icon + Behind badge
• the poll clock (a repeating timer)
• hand over the GitHub token (Keychain)
• read the one Manifest file off disk
↳ never classifies, never decides
Rule of thumb: if it's a decision, it's in TS. If it touches the OS (timer, Keychain, disk, tray), it's a narrow Rust command. The seam is a handful of wires, listed in §4.

3 · The trick that makes it work: a hidden webview

Problem: the logic must run in the webview (§2), but the app has no visible window — it's a menu-bar app. Solution: create the window once at startup, keep it hidden, never close it. It becomes an invisible worker running the engine in the background.

macOS would normally suspend a hidden webview after ~5 min (freezing its timers) to save battery — which would kill the background poll. So the window sets:

"visible": false  ·  "backgroundThrottling": "disabled"

…in tauri.conf.json. That keeps the hidden webview awake. It needs macOS 14+ (below that the flag is ignored). This single assumption holds the whole architecture up — which is exactly why it was spiked first, before building anything on top. That spike is PR #7. (ADR-0005.)

Because the webview is always hidden, the actual UI can't be a web page. The Skill list is drawn as a native macOS tray menu (NSMenu) instead — TS builds the menu items, the OS draws them. (ADR-0009.)

4 · The flow, I/O → presentation (the full intended pipeline)

One poll cycle, end to end. Colors show which side runs each step.

Rust · clock
A repeating timer (tokio interval, every POLL_SECS, default 60s) fires a poll-tick event across the bridge.
TS · listens
The hidden webview hears poll-tick and starts a cycle. (It also polls once on mount, to cover the startup race where the tick beats the listener.)
Rust → TS · the inputs
TS asks Rust for two things it can't reach itself: the Manifest (read_manifest → ~/.agents/.skill-lock.json) and the GitHub token (get_token → Keychain, else gh auth token, else none = slower unauth polling).
TS · derive the work-list
deriveWatchedRepos() groups the Manifest's Skills by Source Repo. One repo + its installed Skills = one Watched Repo, rebuilt fresh each cycle.
TS · poll + classify
assembleSkillStatuses() fetches each repo's branch HEAD, then for every Skill compares its installed folder-tree hash against GitHub and assigns one state (see §5). A baseline cache avoids re-walking history.
TS · cache the result
Results + resolved baselines are written to the app-private store (JSON on disk) so the menu can paint instantly next open. Pure scratch — losing it costs one poll, never data. (ADR-0008.)
TS → Rust · present
TS rebuilds the native tray menu from the statuses and calls set_badge with the Behind count. The tray title shows the badge; the menu lists each Skill (git-helper · 3 behind).

5 · How "Behind" is decided

The Manifest stores each Skill's folder-tree hash at install time — a fingerprint of the folder's contents, not a commit SHA (verified: /git/trees/{hash} resolves, /commits/{hash} 404s — ADR-0003). So comparison is folder-content vs. folder-content, and the count is in Watched Commits — only commits that actually touched that Skill's folder.

"Behind by N" is located by walking the folder's commit history to find the commit that produced the installed hash (the baseline), then counting Watched Commits from there to HEAD. That walk is the expensive bit — so its result is cached, keyed by the immutable hash. (ADR-0004.)
